# event-driven-streaming-platform Constitution

## Core Principles

### I. Event contracts and boundaries

Every feature that publishes or consumes events MUST define stable contracts: event name,
payload shape, and versioning or evolution strategy. Topic names, routing keys, and
exchange bindings MUST be explicit in design docs (plan/spec) before implementation.
Breaking changes MUST use a backward-compatible migration path (e.g., new topic or
schema version) or be documented as an intentional breaking release with consumer
coordination.

**Rationale**: Unbounded event shapes and implicit routing are the main source of
production defects in event-driven systems.

### II. Reproducible runtime and configuration

Workshop and demo environments MUST be reproducible from version-controlled artifacts
(manifests, Helm charts, Compose files, or equivalent). Environment-specific values
MUST be separated from code (config maps, env files not committed with secrets).
`README.md` and epic delivery order in `TODO.md` SHOULD stay aligned with what the
repo actually supports.

**Rationale**: The project teaches operations and consistency; ad-hoc local setup
undermines learning and review.

### III. Testing and failure behavior

Distributed paths (produce → broker → consume, CDC → Kafka, queue publish/consume)
MUST have verifiable tests or scripted checks appropriate to the stack (integration
tests, contract tests on schemas, or documented manual verification in `quickstart.md`).
Consumers that can see duplicates or retries MUST be idempotent or explicitly document
why not.

**Rationale**: Event-driven code fails in ways unit tests alone rarely catch.

### IV. Observability (NON-NEGOTIABLE)

Services and integration components MUST emit structured logs suitable for correlation
where multiple hops exist. Features that touch Kafka, Debezium, or RabbitMQ MUST
consider operational signals called out in the backlog (e.g., consumer lag, connector
health, queue depth) and capture how they will be observed in plan/spec before merge.

**Rationale**: The workshop explicitly targets resilience and operations; observability
is part of the deliverable, not an afterthought.

### V. Data consistency and delivery guarantees

When persisting business data and emitting events, implementations MUST avoid unsafe
dual writes. Prefer transactional outbox or CDC patterns consistent with the
architecture in scope; document at-least-once vs exactly-once expectations and
downstream handling.

**Rationale**: Ties learning goals (Debezium, outbox) to enforceable design rules.

## Platform and stack constraints

This repository targets an event-driven platform on Kubernetes (or equivalent
container runtime) using Apache Kafka for streaming, RabbitMQ for messaging,
PostgreSQL for persistence, and Debezium where change data capture is in scope.
Plans and specs MUST not silently substitute other brokers or databases without an
explicit amendment to this constitution or a recorded complexity justification in
`plan.md` Complexity Tracking.

## Design workflow and artifacts

Feature work SHOULD follow Spec Kit artifacts under `specs/[###-feature-name]/`:
`spec.md`, `plan.md`, and `tasks.md` as generated by the project’s Specify commands.
Backlog themes and epic order remain visible in `TODO.md` at the repository root.

## Governance

This constitution supersedes informal conventions when they conflict. Amendments
require updating `.specify/memory/constitution.md`, bumping **Version** per semantic
rules below, setting **Last Amended** to the amendment date, and propagating any
new non-negotiables to templates (`plan-template.md`, `spec-template.md`,
`tasks-template.md`) and `README.md` when user-facing guidance changes.

**Versioning**: MAJOR — removal or incompatible redefinition of a principle or
constraint. MINOR — new principle, section, or materially expanded requirement.
PATCH — wording, typos, or clarifications that do not change compliance meaning.

**Compliance**: Authors of `plan.md` MUST complete the Constitution Check gates before
Phase 0 research and re-check after Phase 1 design. Reviewers SHOULD reject changes that
violate MUST rules here unless `plan.md` documents an approved exception in Complexity
Tracking.

Runtime orientation for contributors: see `README.md` and `TODO.md`.
